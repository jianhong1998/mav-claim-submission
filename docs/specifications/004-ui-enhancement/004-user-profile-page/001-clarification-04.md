## ISSUE #1: PostgreSQL NULLS Constraint Terminology

**Problem**: Conflicting terminology for NULL handling in unique constraint.

**From Clarification-02, Q2 Answer**:

> "Add `deleted_at` into the composite unique key. Ensure there is `null distinct` implemented."

**From Clarification-03, Conflict #1 Code**:

```ts
@Index(['userId', 'emailAddress', 'deletedAt'], {
  unique: true,
  nulls: 'distinct',  // PostgreSQL: treat NULL values as distinct
})
```

**PostgreSQL Terminology (v15+)**:

| Setting                    | Meaning             | Behavior with NULL                                 |
| -------------------------- | ------------------- | -------------------------------------------------- |
| `NULLS DISTINCT` (default) | NULLs are different | Multiple rows with `deletedAt=null` **ALLOWED** ❌ |
| `NULLS NOT DISTINCT`       | NULLs are equal     | Only ONE row with `deletedAt=null` **ALLOWED** ✓   |

**Desired Behavior** (from business rules):

- ✓ Only ONE active email preference per (userId, emailAddress) combination
- ✓ Multiple deleted rows with different deletedAt timestamps are allowed (history)

**To achieve this, we need `NULLS NOT DISTINCT`**, not `NULLS DISTINCT`.

**Example proving the issue**:

**With `NULLS DISTINCT` (wrong)**:

```sql
-- Both inserts succeed! ❌
INSERT INTO user_email_preferences (userId, emailAddress, type, deletedAt)
VALUES ('user-123', 'personal@gmail.com', 'cc', NULL);

INSERT INTO user_email_preferences (userId, emailAddress, type, deletedAt)
VALUES ('user-123', 'personal@gmail.com', 'bcc', NULL);  -- No error! Two active rows exist!
```

**With `NULLS NOT DISTINCT` (correct)**:

```sql
-- First insert succeeds ✓
INSERT INTO user_email_preferences (userId, emailAddress, type, deletedAt)
VALUES ('user-123', 'personal@gmail.com', 'cc', NULL);

-- Second insert FAILS ✓
INSERT INTO user_email_preferences (userId, emailAddress, type, deletedAt)
VALUES ('user-123', 'personal@gmail.com', 'bcc', NULL);
-- ERROR: duplicate key value violates unique constraint
```

**TypeORM Implementation**:

TypeORM doesn't have direct support for `NULLS NOT DISTINCT` in decorators (as of v0.3.x).

**Option A**: Use raw SQL in migration

```ts
// In migration file
public async up(queryRunner: QueryRunner): Promise<void> {
  await queryRunner.query(`
    CREATE UNIQUE INDEX "IDX_user_email_pref_unique"
    ON "user_email_preferences" ("userId", "emailAddress", "deletedAt")
    NULLS NOT DISTINCT
  `);
}
```

**Option B**: Use TypeORM synchronize (may not support NULLS NOT DISTINCT)

```ts
@Entity('user_email_preferences')
@Index(['userId', 'emailAddress', 'deletedAt'], { unique: true })
// TypeORM will use default NULLS DISTINCT ❌
```

**Option C**: Add WHERE clause to unique index (workaround for older PostgreSQL)

```ts
// In migration file
public async up(queryRunner: QueryRunner): Promise<void> {
  // Partial unique index: only enforce uniqueness where deletedAt IS NULL
  await queryRunner.query(`
    CREATE UNIQUE INDEX "IDX_user_email_pref_active_unique"
    ON "user_email_preferences" ("userId", "emailAddress")
    WHERE "deletedAt" IS NULL
  `);

  // Regular index for foreign key performance
  await queryRunner.query(`
    CREATE INDEX "IDX_user_email_pref_userId"
    ON "user_email_preferences" ("userId")
  `);
}
```

**My recommendation**: **Option C** (partial unique index with WHERE clause)

- Works on all PostgreSQL versions (9.0+)
- Explicit and clear intent
- Ensures only one active row per (userId, emailAddress)
- Allows multiple deleted rows with different timestamps
- TypeORM-friendly (just use raw SQL in migration)

**Question**: Confirm we should use:

1. Partial unique index on `(userId, emailAddress)` WHERE `deletedAt IS NULL` (Option C)
2. Regular index on `userId` for FK performance

Instead of:

- Unique index on `(userId, emailAddress, deletedAt)` with NULLS handling

### Answer

Should still go with Option A. Note that the migration file should be generated by using typeorm command (`make db/migration/generate path="something"`). Then manually update the migration script to include the `NULL NOT DISTINCT`.

## ISSUE #2: Multiple Deleted Rows Edge Case

**Problem**: Restore logic is undefined when multiple soft-deleted rows exist for the same email.

**Scenario that creates multiple deleted rows**:

If implementation has a bug or during data migration, we might end up with:

```
user_email_preferences table:
┌──────────┬──────┬──────────────────────┬─────────────────────┐
│ id       │ type │ emailAddress         │ deletedAt           │
├──────────┼──────┼──────────────────────┼─────────────────────┤
│ abc-123  │ cc   │ personal@gmail.com   │ 2025-11-10 10:00:00 │ ← Deleted 3 days ago
│ def-456  │ bcc  │ personal@gmail.com   │ 2025-11-12 15:30:00 │ ← Deleted 1 day ago (most recent)
│ ghi-789  │ cc   │ personal@gmail.com   │ null                │ ← Currently active
└──────────┴──────┴──────────────────────┴─────────────────────┘
```

Wait, with proper unique constraint, row `ghi-789` (active) prevents other active rows. But multiple deleted rows CAN exist.

**More realistic scenario** (if unique constraint is missing or buggy):

```
┌──────────┬──────┬──────────────────────┬─────────────────────┐
│ id       │ type │ emailAddress         │ deletedAt           │
├──────────┼──────┼──────────────────────┼─────────────────────┤
│ abc-123  │ cc   │ personal@gmail.com   │ 2025-11-10 10:00:00 │ ← Deleted (older)
│ def-456  │ bcc  │ personal@gmail.com   │ 2025-11-12 15:30:00 │ ← Deleted (newer)
└──────────┴──────┴──────────────────────┴─────────────────────┘
```

## ISSUE #3: Clarification on Smart Diff Algorithm

**Context**: Based on Clarification-02 Q6 answer (Option B - Smart diff) and Clarification-03 Conflict #2/3 answers (allow type switching).

**The complete smart diff algorithm should be**:

**For each email in NEW preference list**:

1. Find ALL existing rows for `(userId, emailAddress)` including soft-deleted
2. **If ACTIVE row exists**:
   - Compare type
   - If type matches: SKIP (no change)
   - If type differs: UPDATE type field (Conflict #3, Case 2)
3. **If ONLY deleted rows exist**:
   - Restore one row (see Issue #2 for which one)
   - Update type if needed (Conflict #2 Option A)
4. **If NO rows exist**:
   - CREATE new row

**For each ACTIVE email preference NOT in new list**:

1. SOFT DELETE

**Transaction boundaries**:

- Wrap entire operation in database transaction
- Use TypeORM transaction manager
- Rollback on any failure

**Example implementation**:

```ts
async updateUserEmailPreferences(
  userId: string,
  newPreferences: { type: 'cc' | 'bcc'; emailAddress: string }[]
): Promise<void> {
  await this.dataSource.transaction(async (manager) => {
    const repo = manager.getRepository(UserEmailPreferenceEntity);

    // Get all current active preferences
    const currentActive = await repo.find({ where: { userId } });

    // Process each new preference
    for (const newPref of newPreferences) {
      const allRows = await repo.find({
        where: { userId, emailAddress: newPref.emailAddress },
        withDeleted: true,
      });

      const activeRow = allRows.find(row => row.deletedAt === null);
      const deletedRows = allRows.filter(row => row.deletedAt !== null);

      if (activeRow) {
        // Update type if different
        if (activeRow.type !== newPref.type) {
          activeRow.type = newPref.type;
          await repo.save(activeRow);
        }
      } else if (deletedRows.length > 0) {
        // Restore most recent deleted row
        const toRestore = deletedRows.sort((a, b) =>
          b.deletedAt.getTime() - a.deletedAt.getTime()
        )[0];
        await repo.restore(toRestore.id);

        // Update type if different
        if (toRestore.type !== newPref.type) {
          toRestore.type = newPref.type;
          await repo.save(toRestore);
        }
      } else {
        // Create new row
        const newRow = repo.create({
          userId,
          type: newPref.type,
          emailAddress: newPref.emailAddress,
        });
        await repo.save(newRow);
      }
    }

    // Soft delete removed preferences
    const newEmails = new Set(newPreferences.map(p => p.emailAddress));
    const toDelete = currentActive.filter(row => !newEmails.has(row.emailAddress));

    for (const row of toDelete) {
      await repo.softRemove(row);
    }
  });
}
```

2. Should we validate CC/BCC cross-type constraint before saving (from Clarification-01 Q4B)?
3. Should we send test email for newly added preferences during bulk update?
